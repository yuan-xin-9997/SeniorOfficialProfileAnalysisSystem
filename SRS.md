## 1. 引言

### 1.1 目的

本文档为"高级官员履历分析系统"（以下简称"本系统"或"SOPAS"）的软件需求规格说明书（SRS），旨在明确系统的功能需求、非功能需求、数据模型、接口规范及技术架构，为后续的设计、开发和测试提供依据。

### 1.2 项目范围

本系统用于采集、存储、分析和可视化展示中国共产党中央委员会委员及候补委员（以下简称"高级官员"）的公开履历信息，并通过关系分析算法挖掘官员之间的潜在关联。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 高级官员 | 中国共产党中央委员会委员及候补委员 |
| 履历 | 官员从出生至今的逐年经历记录 |
| 同盟关系 | 两位或多位官员因曾在同一地点/部门、同一时间段共事而形成的关联关系 |
| 关系强度 | 量化指标，衡量两位官员之间关联的紧密程度 |
| 数据源 | 政府官方网站，如人民网、新华网、中国政府网等 |

### 1.4 参考文档

- IEEE 830-1998 软件需求规格说明标准
- 人民网 - 中国党政领导干部资料库
- 新华网 - 领导人资料库
- 中国政府网 - 国务院及各部门领导简历

## 2. 总体描述

### 2.1 系统概述

本系统是一个基于 Python 全栈技术构建的 Web 应用，包含四大核心模块：

```
┌─────────────────────────────────────────────────────────┐
│                    高级官员履历分析系统                      │
├──────────────┬──────────────┬─────────────┬─────────────┤
│  数据抓取模块  │  数据存储模块  │ 数据分析模块  │ 数据展示模块  │
│  (Scraper)   │  (Storage)   │ (Analysis)  │ (Display)   │
└──────────────┴──────────────┴─────────────┴─────────────┘
```

### 2.2 用户角色

| 角色 | 描述 | 权限 |
|------|------|------|
| 管理员 | 系统管理者 | 全部功能，包括用户管理、抓取任务配置、数据管理 |
| 普通用户 | 数据查看与分析者 | 查看数据、执行分析、查看可视化，不可管理用户和系统配置 |

### 2.3 运行环境

- **部署方式：** Docker + Kubernetes (K8S)
- **服务端：** Python 后端服务
- **前端：** 现代 Web 前端框架
- **数据库：** 图数据库（主）+ 关系型数据库（辅）
- **运行平台：** Linux 容器环境

### 3.3 模块三：数据分析模块（Analysis）

#### 3.3.1 功能概述

基于存储的履历数据和关系图谱，提供多维度的分析能力，核心功能为官员间"同盟"关系的识别与评估。

#### 3.3.2 功能需求列表

| 编号 | 功能 | 优先级 | 描述 |
|------|------|--------|------|
| ANA-001 | 同盟关系识别 | P0 | 基于共同履历识别官员之间的潜在同盟关系 |
| ANA-002 | 关系强度计算 | P0 | 综合多维度计算两位官员之间的关系强度评分 |
| ANA-003 | 关系路径查询 | P0 | 查询两位官员之间的关系链路（直接/间接关系） |
| ANA-004 | 派系聚类分析 | P1 | 基于关系网络自动识别潜在的派系/圈子 |
| ANA-005 | 官员相似度分析 | P1 | 比较两位官员的履历相似度 |
| ANA-006 | 晋升路径分析 | P1 | 分析官员的典型晋升路径和模式 |
| ANA-007 | 地域分布分析 | P2 | 分析官员的地理分布特征 |
| ANA-008 | 年龄梯队分析 | P2 | 分析官员的年龄梯队和代际分布 |
| ANA-009 | 分析报告生成 | P2 | 自动生成分析报告（PDF/HTML） |

#### 3.3.3 同盟关系判定算法

**判定维度：**

| 维度 | 权重 | 判定规则 |
|------|------|----------|
| 同一部门共事 | 0.30 | 在同一部门任职且时间有重叠 |
| 同一地区任职 | 0.25 | 在同一省/市/县任职且时间有重叠 |
| 上下级关系 | 0.20 | 存在直接的上下级汇报关系 |
| 校友关系 | 0.10 | 同一院校毕业 |
| 同乡关系 | 0.10 | 出生地或籍贯相同 |
| 时间重叠度 | 0.05 | 共事时间越长，关系越强 |

**关系强度计算公式：**

```
Strength = Σ(Wi × Si) × TimeFactor

其中：
- Wi = 第 i 个维度的权重
- Si = 第 i 个维度的匹配得分（0 或 1）
- TimeFactor = 时间重叠系数（重叠年数 / 最大可能年数）
- 最终 Strength ∈ [0, 1]
```

**同盟判定阈值：**

| 关系强度区间 | 关系等级 | 描述 |
|-------------|---------|------|
| 0.8 - 1.0 | 密切同盟 | 多维度深度关联 |
| 0.5 - 0.8 | 一般同盟 | 存在明确共事经历 |
| 0.3 - 0.5 | 弱关联 | 存在间接关联 |
| 0.0 - 0.3 | 无明显关联 | 关联度较低 |

#### 3.3.4 分析查询接口

```
# 查询两位官员之间的关系
GET /api/analysis/relationship?official_a={id}&official_b={id}

# 查询某官员的所有关联官员
GET /api/analysis/connections?official_id={id}&min_strength=0.3

# 查询两位官员之间的关系路径
GET /api/analysis/path?from={id}&to={id}&max_depth=3

# 执行派系聚类分析
POST /api/analysis/clustering

# 查询官员履历相似度
GET /api/analysis/similarity?official_a={id}&official_b={id}
```

| DSP-003 | 地理分布图 | P0 | 在中国地图上展示官员任职地理分布 |
| DSP-004 | 统计图表 | P1 | 年龄分布、学历分布、地域分布等统计图 |
| DSP-005 | 官员详情页 | P0 | 展示单个官员的完整信息和关系 |
| DSP-006 | 关系对比页 | P1 | 并排对比两位官员的履历和关系 |
| DSP-007 | 搜索与筛选 | P0 | 按姓名、地区、职位、届次等条件搜索筛选 |
| DSP-008 | 派系可视化 | P1 | 以不同颜色/分组展示识别出的派系 |
| DSP-009 | 数据仪表盘 | P2 | 系统概览仪表盘，展示关键统计数据 |
| DSP-010 | 响应式布局 | P2 | 适配不同屏幕尺寸 |

#### 3.4.3 可视化规格

**关系网络图：**
- 类型：力导向图（Force-Directed Graph）
- 节点：代表官员，大小表示行政级别，颜色表示派系/状态
- 边：代表关系，粗细表示关系强度，颜色表示关系类型
- 交互：支持拖拽、缩放、点击查看详情、高亮关联路径
- 技术：D3.js 或 ECharts

**履历时间线：**
- 类型：水平/垂直时间轴
- 内容：按年份展示教育、从政、军旅等经历
- 交互：支持缩放、点击展开详情
- 技术：D3.js 或自定义组件

**地理分布图：**
- 类型：中国省级行政区划热力图
- 内容：展示官员在各省份的任职分布
- 交互：支持点击省份查看详情、时间维度播放
- 技术：ECharts Map 或 DataV

**统计图表：**
- 类型：柱状图、饼图、散点图等
- 内容：年龄分布、学历分布、地域分布、届次分布
- 交互：支持筛选、下钻
- 技术：ECharts 或 Chart.js

## 9. 约束与假设

### 9.1 约束

1. **数据合规性：** 系统仅采集公开可获取的政府公开信息，不涉及任何非公开或敏感数据
2. **爬虫合规性：** 遵守目标网站的 robots.txt 协议，控制抓取频率，不对目标网站造成负担
3. **数据准确性：** 系统不保证抓取数据的 100% 准确性，需要人工审核确认
4. **网络依赖：** 数据抓取功能依赖外部网络访问目标网站

### 9.2 假设

1. 目标数据源网站在系统运行期间保持可访问
2. 目标数据源网站的页面结构在短期内不会发生重大变化
3. 系统用户具备基本的计算机操作能力
4. 部署环境具备稳定的网络和计算资源
